# Спек: секция «Бюджет» на странице «окружение» (`docs/mirea-list.html`)

**Дата:** 2026-07-15
**Статус:** дизайн утверждён, готов к плану реализации
**Автор:** brainstorming-сессия (Claude + Денис)
**Связано:** [[2026-07-15-neighbors-paid-filter-design]] (платная секция «окружение»),
[[2026-07-15-mirea-neighbors-list-design]] (базовая фича)

## 1. Проблема и цель

Страница «окружение» (`docs/mirea-list.html`) сейчас показывает **одну** секцию — платный
конкурс «1. Интеллектуальные системы…» (`track_neighbors: true` только на нём), фильтр по
`consent` («соблюдены условия для платного»). Пользователь хочет **так же отслеживать свою
позицию на бюджете** того же направления (конкурс `1862638623056207158`, `edu_level=2`).

На официальном сайте МИРЭА для бюджета есть фильтр **«Проходной ВП»**: при включении список
показывает только проходящих и перенумеровывает их сквозняком 1..N. Именно этот состав нужен.

**Цель.** На той же странице «окружение» появляется **вторая секция** — бюджетный конкурс,
показывающая **только** абитуриентов с «Проходной ВП: да», перенумерованных 1..N. В конкурсе
`1862638623056207158` это ровно 11 человек (= число мест), код `1366129` — №6.

## 2. Ключевая находка (фильтр бюджета ≠ фильтр платного)

Официальный фильтр «Проходной ВП» на бюджете — это поле **`iHPO`** в API
(`/competitions_api/entrants`), в модели **`Entrant.passing_real`**. НЕ `accepted`/`consent`.

**Доказательство** (живой API конкурса `1862638623056207158`, 2026-07-15 15:06):

- `iHPO=1` ровно у **11** человек (= `plan`/число мест). Набор кодов **точно** совпадает со
  скриншотом офсайта (тот же состав, тот же порядок), код `1366129` — №6.
- `accepted=1` у **84** человек выше нашего места — фильтр по `consent` дал бы не тот
  (слишком большой) список. Поэтому для бюджета фильтруем по `passing_real`, а не `consent`.
- Все 11 из `iHPO`-набора здесь также имеют `accepted=1` и `isActive=1`, но фильтрующее
  условие — именно `passing_real`.

> Терминология (см. [[2026-07-15-neighbors-paid-filter-design]] §2): в модели
> `Entrant.consent` ← API `accepted`; `Entrant.passing_real` ← API `iHPO` (Проходной ВП);
> `Entrant.passing_main` ← API `iHP` (Основной ВП). `iHPO` — приоритетно-согласиевый прогноз
> МИРЭА: ровно `plan` человек, кто реально зачислится после разрешения приоритетов по всем
> конкурсам; сырое `place` (у нас 1688) к нему не привязано.

## 3. Что уже есть (не переизобретаем)

- **Инфраструктура секций уже мультиконкурсная.** `_gather_neighbors` (`dashboard.py`)
  итерирует **все** watches с `track_neighbors=true` и возвращает spec-словарь на каждый;
  `build_neighbors_html` рендерит по секции на spec через `_neighbor_section`. Добавление
  второго `track_neighbors` watch автоматически даёт вторую секцию — новой страницы/навигации
  не требуется.
- **Данные уже сохраняются.** Каждый снапшот в `state.db` содержит всех `Entrant` с полями
  `place`, `consent`, `passing_real`, `passing_main`, `is_active`, баллами. Новый сетевой
  запрос не нужен — фильтруем в памяти.
- **Тип конкурса уже определяется.** `spec["paid"] = is_paid(title) or is_paid(group/name)`;
  `is_paid` (`format.py:52`) = «договор»/«платн» в тексте. Бюджетный watch (title «…/ общий»,
  группа «МИРЭА — бюджет») → `paid=False`; платный (title «…/ договор») → `paid=True`.
- **Сортировка/нумерация уже правильные.** Сортировка по `place`, сквозная нумерация 1..N в
  `_neighbor_section` (`enumerate(rows, 1)`). Для 11 бюджетных порядок по `place` = порядок по
  баллам убыв. = порядок офсайта. Ничего нового считать не нужно.

## 4. Решения (утверждены в брейнсторме)

| Вопрос | Решение |
|---|---|
| Фильтр бюджета | **`passing_real` (= iHPO)**, не `consent`. Плюс `is_active`. Платный фильтр (`consent`) без изменений. |
| Выбор фильтра | По `spec["paid"]`: `consent` если платный, иначе `passing_real`. |
| Структура | **Вариант A** — та же страница `mirea-list.html`, **две секции** (Платно + Бюджет). Без отдельной страницы/навигации (YAGNI). |
| Порядок секций | По порядку watches в `config.yaml` (бюджетный раньше → секция «Бюджет» выше). Переставляется одной строкой конфига. |
| Метка секции | В заголовок `<h2>` добавляем «Бюджет · » / «Платно · » для различения. |
| Колонка-флаг бюджета | Остаётся «Согл» (= `consent`, офсайтовая «Наличие согласия»). Не дублирует «Прох.ВП», который уже виден в «Примечании». `_neighbor_row` не меняется. |
| Наша строка отсутствует | Если наш `passing_real=0` на бюджете — полный отфильтрованный список + баннер (адаптированный текст, см. §6). |

## 5. Архитектура (изменения в `dashboard.py` + `config.yaml`)

### 5.1 `config.yaml` (приватный, gitignored)

Добавить `track_neighbors: true` на бюджетный watch «1. Интеллектуальные системы управления и
обработки информации» (группа «МИРЭА — бюджет», конкурс `1862638623056207158`).

### 5.2 `_gather_neighbors` — фильтр по типу конкурса

Вычислить `paid` **до** фильтрации, затем выбрать предикат:

```python
paid = is_paid(title) or is_paid(w.group or w.name)
ok = (lambda e: e.consent) if paid else (lambda e: e.passing_real)
eligible = sorted(
    [e for e in snap.entrants if e.place is not None and e.is_active and ok(e)],
    key=lambda e: e.place,
)
we_absent = not any(e.code in our_codes for e in eligible)
```

Остальные поля spec (`title`, `updated_at`, `fetched_at`, `paid`, `our_codes`, `rows=eligible`)
— как сейчас.

### 5.3 `_neighbor_section` — параметризация текстов по `paid`

Строки, «зашитые» под платно, разветвляем по `spec["paid"]`:

- **Заголовок:** `<h2>` получает префикс-метку. `paid` → «Платно · {title}», иначе «Бюджет · {title}».
- **Пустой список:** `paid` → «Пока никто не выполнил условия для платного.»;
  иначе → «Пока никто не проходит по Проходному ВП.».
- **Баннер отсутствия:** `paid` → «вашего кода нет среди выполнивших условия для платного»;
  иначе → «вашего кода нет среди проходящих по Проходному ВП».

Колонка-флаг и её заголовок (`flag_hdr = "Платн" if paid else "Согл"`, `flag = e.consent`),
подсветка строк, «Примечание» (`_note`), сквозная нумерация — **без изменений**.

### 5.4 `build_neighbors_html` — докстринг и футер

- Докстринг: описать оба фильтра (платно=`consent`, бюджет=`passing_real`).
- Футер: «один конкурс» → «конкурсы» (или убрать число).

## 6. Крайние случаи

- **`eligible` пуст** (бюджет) → секция с «Пока никто не проходит по Проходному ВП.», без падений.
- **Наш код в конкурсе, но `passing_real=0`** (бюджет) → `we_absent=True`, показываем полный
  отфильтрованный список из 11 + баннер «вашего кода нет среди проходящих по Проходному ВП»,
  наша строка не подсвечена как «вы».
- **Платная секция** — поведение полностью прежнее (регрессия должна быть зелёной).

## 7. Что НЕ меняется

Топбар и ссылки, деплой (`publish-dashboard.sh`), подсветка строк, колонка «Примечание»,
сортировка по `place`, сквозная нумерация, условие существования страницы (`has_neighbors`),
стили, `_neighbor_row`. Правка изолирована в `_gather_neighbors`, `_neighbor_section`,
`build_neighbors_html` (докстринг/футер) + одна строка `config.yaml`.

## 8. Тесты (TDD, `tests/test_neighbors.py`)

Новые/обновлённые кейсы на синтетическом снапшоте:

1. **Бюджет фильтрует по `passing_real`:** entrant с `passing_real=1, consent=0` появляется;
   с `passing_real=0, consent=1` — не появляется (обратное платному фильтру).
2. **Бюджет + `is_active`:** `passing_real=1, is_active=False` исключается.
3. **Платный фильтр не сломан:** платный watch по-прежнему фильтрует по `consent`
   (`consent=1, passing_real=0` появляется; `consent=0, passing_real=1` — нет).
4. **Метка секции:** бюджетная секция содержит «Бюджет», платная — «Платно».
5. **Тексты бюджета:** пустой список → «Пока никто не проходит по Проходному ВП»; баннер
   отсутствия → «…проходящих по Проходному ВП» (без «для платного»).
6. **Две секции на одной странице:** при двух `track_neighbors` watches (платный+бюджетный)
   `render_pages` даёт один `mirea-list.html` с обеими секциями.
7. **Сквозная нумерация бюджета:** № = 1..N по порядку `place`, наша строка помечена «◄ вы».

Регрессия: весь набор `pytest` зелёный (был 142/142).

## 9. Проверка end-to-end

1. `python -m pytest tests/test_neighbors.py -q` — новые кейсы зелёные.
2. `python -m pytest -q` — регрессий нет.
3. `python -m vuz_monitor dashboard` на реальном `state.db` → в `docs/mirea-list.html` две
   секции; бюджетная содержит ровно 11 (`passing_real=1`), сквозная нумерация 1..11,
   код `1366129` — №6.
4. Открыть `docs/mirea-list.html` в браузере → сверить состав и порядок бюджетной секции со
   скриншотом офсайта (1289372, 1640958, 1503592, 1687874, 1726973, 1366129, …).
