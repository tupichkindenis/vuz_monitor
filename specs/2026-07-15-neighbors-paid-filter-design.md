# Спек: фильтр «соблюдены условия для платного» на странице «окружение» (`docs/mirea-list.html`)

**Дата:** 2026-07-15
**Статус:** дизайн утверждён, готов к плану реализации
**Автор:** brainstorming-сессия (Claude + Денис)
**Связано:** [[2026-07-15-mirea-neighbors-list-design]] (базовая фича «окружение»)

## 1. Проблема и цель

Страница «окружение» (`docs/mirea-list.html`) сейчас показывает **всех** абитуриентов в
окне «все на нашем месте и выше + 10 после». Для платного конкурса это шум: большинство
из списка просто подали документы по баллам и не займут платное место. Нужны только те,
кто реально закрепился на платном.

На официальном сайте МИРЭА (`/accepted-entrants-list/competition?comp_ids=…`) есть чекбокс
**«Соблюдены условия для платного»**: при его включении таблица показывает только
подходящих абитуриентов и **перенумеровывает их сквозняком 1..N**. Именно этот состав нужен.

**Цель.** Страница «окружение» для платного конкурса показывает **только** абитуриентов с
«соблюдены условия для платного», перенумерованных сквозняком 1..N, как отфильтрованный
вид на офсайте. В конкурсе `1862638623058304310` это ровно 14 человек, код `1366129` — №1.

## 2. Ключевая находка (важнее самого фильтра)

Фильтр «Соблюдены условия для платного» на офсайте — это поле **`accepted`** в API
(`/competitions_api/entrants`), а НЕ поле `pc`.

**Доказательство** (живой API конкурса `1862638623058304310`, 2026-07-15):

- Код `1364851`: `accepted=1`, `pc=0`, `iHP=0`, `iHPO=0` — на офсайте «да». Единственное
  поле = 1 у него — `accepted`.
- По всем 7 строкам скриншота колонка «да» ⟺ `accepted=1`. По `pc=1` — другие 73 человека,
  нашего кода среди них нет.
- `accepted=1 AND isActive=1` (14 человек), сортировка по `place`, сквозная нумерация →
  **точно воспроизводит** порядок офсайта (`1366129, 1179201, 1569330, …`).
- Наш же код это уже подтверждает: `notify.py:63` и карточки `dashboard.py:330` для подписи
  «Соблюдены условия для платного» берут `consent` (= `accepted`).

**Скрытый баг.** Страница «окружение» (`_neighbor_row`, `dashboard.py:917`) показывает
`e.paid_ok` (= `pc`) как флаг для платных конкурсов. Это не то поле. Правильно — `e.consent`
(= `accepted`). Мислейбл тянется из докстринга адаптера (`mirea_api.py:14`) и базовой спеки
neighbors. Здесь исправляем.

> Терминология: в модели `Entrant.consent` ← API `accepted`; `Entrant.paid_ok` ← API `pc`.
> Для **платного** конкурса `accepted` = «заключил и оплатил договор» = «соблюдены условия
> для платного». Поле `pc` — не то, что показывает офсайтовый фильтр; в этой фиче не используется.

## 3. Что уже есть (не переизобретаем)

- **Данные уже сохраняются.** Каждый снапшот в `state.db` содержит всех `Entrant` с полями
  `place`, `consent` (= accepted), `is_active`, `paid_ok`, `passing_main`/`passing_real`,
  `priority`, баллами. Новый сетевой запрос и парсинг не нужны — фильтруем в памяти.
- **Окружение уже строится** — `_gather_neighbors` (`dashboard.py:130`) → список spec-словарей,
  `build_neighbors_html` рендерит `mirea-list.html`. Меняем эти две зоны, не трогая пайплайн/деплой.
- **Хелперы** `esc`, `g`, `yesno`, `is_paid` в `format.py` — переиспользуем.

## 4. Решения (утверждены в брейнсторме)

| Вопрос | Решение |
|---|---|
| Поле фильтра | **`consent` (= accepted)**, не `paid_ok`. Заодно чиним колонку-флаг neighbors. |
| Доп. условие | **`is_active`** (офсайтовый «Только активные»). Среди подходящих это почти всегда True, берём для точного совпадения. |
| Состав списка | **Вариант A — весь отфильтрованный список** (все подходящие), без окна «10 ниже». Набор мал (реально оплатившие). |
| Нумерация «№» | **Сквозная 1..N** по отфильтрованному списку. Официальное `place` **не** показываем — ровно как на офсайте. |
| Охват | Фильтр **включён всегда** для страницы «окружение» (единственный `track_neighbors` watch — платный). Без нового конфиг-флага (YAGNI). |
| Наша строка отсутствует | Если наш код не прошёл фильтр — показываем полный отфильтрованный список + баннер (см. §6). |

## 5. Архитектура (изменения только в `dashboard.py`)

### 5.1 `_gather_neighbors` (dashboard.py:130) — фильтр + полный список

- Из `ranked` (все entrants с `place`, сортированы по `place`) построить
  `eligible = [e for e in ranked if e.consent and e.is_active]`.
- `rows = eligible` целиком (без среза `NEIGHBORS_AFTER`).
- `we_absent = (наши коды не встречаются в eligible)`.
- Остальные поля spec (`title`, `updated_at`, `fetched_at`, `paid`, `our_codes`) — как сейчас.
- `NEIGHBORS_AFTER` и логика окна «ahead_and_self + after» в этой ветке больше не применяются.

### 5.2 Рендер (`_neighbor_section` / `_neighbor_row`, dashboard.py:906-954) — сквозной №, флаг = consent

- `_neighbor_section`: при сборке `body` перечислять строки через `enumerate(spec["rows"], 1)`
  и передавать порядковый номер в `_neighbor_row`.
- `_neighbor_row(e, seq, our_codes, paid)`:
  - колонка «№» показывает `seq` (1..N), **не** `e.place`. Пометка «◄ вы» — как сейчас, по `our_codes`.
  - флаг-колонка: `flag = e.consent` (было `e.paid_ok if paid else e.consent`). После фильтра
    там везде «да» — как на офсайте.
- Подсветка строк (`pass-real`/`pass-main`/`you`), колонка «Примечание» (`_note`) — без изменений.

## 6. Крайние случаи

- **`eligible` пуст** → секция показывает «Пока никто не выполнил условия для платного»
  (аналог текущего пустого состояния), без падений.
- **Наш код есть в конкурсе, но не в `eligible`** (`consent=0`) → `we_absent=True`,
  показываем полный отфильтрованный список + баннер «вашего кода нет среди выполнивших условия
  для платного» (адаптируем текст существующего баннера `_neighbor_section`).
- **Бюджетный `track_neighbors` watch** (сейчас таких нет): фильтр по `consent` = «подавшие
  согласие». Поведение консистентно; если понадобится иначе — отдельная задача.

## 7. Что НЕ меняется

Топбар и ссылки, деплой (`publish-dashboard.sh`), подсветка строк, колонка «Примечание»,
условие включения страницы (`has_neighbors`), стили. Правка изолирована в `_gather_neighbors`,
`_neighbor_section`, `_neighbor_row`.

## 8. Тесты (TDD, `tests/test_neighbors.py`)

Новые/обновлённые кейсы на синтетическом снапшоте:

1. **Фильтрует по consent:** entrant с `consent=0` в списке не появляется; с `consent=1` — появляется.
2. **Фильтрует по is_active:** `consent=1, is_active=False` исключается.
3. **Сквозная нумерация:** колонка «№» = 1,2,3… по порядку, а не официальное `place`
   (кейс с «дырявыми» местами, напр. places 5, 40, 800 → №1,2,3).
4. **Полный список:** при >11 подходящих показываются все, без обрезки на 11.
5. **Флаг = consent, не paid_ok:** строка с `consent=1, paid_ok=0` показывает флаг «да».
6. **we_absent при нашем `consent=0`:** баннер присутствует, наш код не подсвечен как «вы».
7. **Пустой eligible:** секция с сообщением, без исключений.

Регрессия: весь набор `pytest` зелёный (был 139/139).

## 9. Проверка end-to-end

1. `python -m pytest tests/test_neighbors.py -q` — новые кейсы зелёные.
2. `python -m pytest -q` — регрессий нет.
3. `python -m vuz_monitor dashboard` (или прямой вызов `render_pages`) на реальном `state.db`
   → в `docs/mirea-list.html` секция платного конкурса содержит ровно подходящих (`consent=1`),
   сквозная нумерация 1..N, код `1366129` — №1.
4. Открыть `docs/mirea-list.html` в браузере (browse) → сверить состав и порядок со
   скриншотом офсайта (первые строки: 1366129, 1179201, 1569330, …).
